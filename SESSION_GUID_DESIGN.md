# Session GUID — WhisperCal side design

**Status:** Implemented (WhisperCal side, 2026-07-17; Tome side also done — see `../Tome/SESSION_GUID_DESIGN.md`). Build + lint clean; the manual testing checklist below has NOT been run yet. Implementation notes vs. the original plan: the unlinked-flow "pre-select in the pick modal" landed as something better — a definitive guid match in `CalendarView.autoLinkObvious` links automatically with no modal at all (`autoLinkBySessionGuid`); `ApiUnlinkedProvider.linkToNote` also mirrors the transcript's `session_guid` onto the note on manual links.

## Problem

Today no identifier crosses the WhisperCal↔Tome boundary. `POST /start` returns bare `{"ok":true}` and linking a finished transcript back to a meeting card rests on:

- polling a single **global** `GET /status` (`waitAndLink`, `src/services/ApiRecording.ts:300-317`) — ambiguous when Tome is transcribing meeting A while recording meeting B (the normal back-to-back case);
- a filename-prefix + newest-file scan (`findNewestFile`, `src/services/ApiRecording.ts:218-233`) — collides when two notes share a basename, breaks if Tome names the file differently than the sent `suggestedFilename`.

Additionally, all in-flight recording state lives in memory only (`CardUiState`), so an Obsidian restart mid-recording orphans the session and requires the manual unlinked-recordings flow.

## Solution overview

WhisperCal generates a GUID at recording start, sends it to Tome, and every artifact and status answer carries it from then on. Linking becomes an id lookup; the filename heuristics remain only as a fallback for an old Tome build.

Decisions already made (do not re-litigate):

1. **GUID is generated by WhisperCal** in `startApiRecording()` — the initiator owns identity. Tome mints its own GUID only for sessions WhisperCal didn't start (manual/menu-bar, voice memos), so the field is uniformly present.
2. **Restart recovery is in scope**: in-flight `guid → note` state is persisted to `data.json` and reconciled on plugin load.
3. **Legacy filename matching stays as a fallback**, tried only after GUID paths fail. Remove in a later release.

## Shared protocol (must match Tome doc verbatim)

**GUID format:** UUIDv4, lowercase, canonical hyphenated form. WhisperCal: `crypto.randomUUID()`. Tome: `UUID().uuidString.lowercased()`.

**Canonical field names:** `sessionGuid` in all JSON (API payloads, voiceprint sidecar); `session_guid` in all Markdown frontmatter (meeting note, transcript).

### `POST /start` request (additive)

```jsonc
{
  "sessionGuid": "3f1c9e2a-8b4d-4c6e-9f0a-2d7b5e8c1a4f",   // NEW, optional
  "suggestedFilename": "2026-07-17 0930 - Standup - Transcript",
  "meetingContext": { "subject": "…", "attendees": ["…"], "calendarEventId": "…", "startTime": "…" }
}
```

### `POST /start` response (was bare `{"ok":true}`)

```jsonc
{ "ok": true, "sessionGuid": "3f1c…", "sessionId": "session_2026-07-17_09-30-01" }
```

`sessionGuid` echoes the supplied value (or Tome's minted one if absent). `sessionId` is Tome's internal timestamp id (informational only — it is second-granular and collidable; never use it as a correlation key).

### `GET /sessions/by-guid/{guid}/status` (new)

```jsonc
// 200
{
  "sessionGuid": "3f1c…",
  "sessionId": "session_2026-07-17_09-30-01",
  "state": "recording" | "transcribing" | "complete" | "failed",
  "startedAt": "2026-07-17T09:30:01-04:00",        // present while recording
  "transcriptFilename": "Standup - Transcript.md",   // present when state == "complete" — FINAL basename after collision -1/-2 suffixes and renames
  "transcriptPath": "/Users/dloomis/Documents/Tome/Meetings/Standup - Transcript.md", // absolute, when complete
  "error": "…"                                       // present when state == "failed"
}
// 404 { "error": "unknown sessionGuid" } — guid never seen, or evicted (Tome retains the last 20 completed sessions)
```

### `GET /status` (global, additive)

`recording` gains `sessionGuid` so the global poll can at least identify *whose* recording is live:

```jsonc
{ "state": "recording", "recording": { "subject": "…", "suggestedFilename": "…", "sessionGuid": "3f1c…" } }
```

### Artifacts (written by Tome)

- Transcript frontmatter gains `session_guid: "3f1c…"` — written by the **live** logger at session start (so even a crash-orphaned transcript carries it) and preserved through finalize.
- Voiceprint sidecar (`*.voiceprints.json`) gains top-level `"sessionGuid": "3f1c…"`. Schema stays `1` (additive optional field; WhisperCal's decoder ignores unknown fields).

### Compatibility matrix

| | old Tome | new Tome |
|---|---|---|
| **old WhisperCal** | today's behavior | no guid supplied → Tome mints one; extra frontmatter/status fields are ignored by old WhisperCal. No breakage. |
| **new WhisperCal** | `/start` ignores the unknown `sessionGuid` field and returns bare `{"ok":true}` → WhisperCal detects the missing echo and runs the **entire legacy flow** for that session | full GUID flow |

## WhisperCal changes

### 1. Constants — `src/constants.ts`

Add next to `MACWHISPER_SESSION_ID` in the `FM` map:

```ts
SESSION_GUID: "session_guid",
```

### 2. Transport — `src/services/RecordingApi.ts`

- `recordingStart(baseUrl, suggestedFilename, meetingContext, sessionGuid)` — add `sessionGuid` to the `POST /start` body. Parse the response defensively; return `{ guidAcknowledged: boolean, sessionId?: string }` where `guidAcknowledged` is true iff the response echoes the same `sessionGuid`. Bare `{"ok":true}` (old Tome) → `guidAcknowledged: false`. Never throw on an unparseable-but-2xx body — treat as unacknowledged.
- New `recordingSessionStatus(baseUrl, sessionGuid)` → `GET /sessions/by-guid/{guid}/status`. Return a typed result: `{ state, transcriptFilename?, transcriptPath?, startedAt?, error? }`. Map HTTP 404 to `{ state: "unknown" }` (distinct from a network error, which should throw like the existing `recordingStatus`).
- Keep the existing global `recordingStatus` untouched (fallback path + `confirmIfServiceRecording` in `MeetingCard.ts`).

### 3. Start flow — `src/services/ApiRecording.ts` (`startApiRecording`, ~line 46)

1. Generate `const sessionGuid = crypto.randomUUID();` before calling `/start`.
2. Pass it to `recordingStart`. On success (2xx):
   - Store `sessionGuid` and `guidAcknowledged` on `RecordingInfo` (see §4).
   - Write `session_guid` to the **note** frontmatter via `batchUpdateFrontmatter` (after the 200, so a failed start never stamps the note). If a previous `session_guid` exists on the note (re-recording a meeting), overwrite it — the note tracks the *latest* session; the transcript file preserves history per-file.
   - Persist the in-flight record to `data.json` (see §7).
3. If `guidAcknowledged` is false, still store the guid (the note keeps it for bookkeeping) but mark the recording as legacy-mode so the watch/link tail skips per-guid polling.

### 4. In-flight state — `src/services/CardUiState.ts`

`RecordingInfo` (lines 4-30) gains:

```ts
sessionGuid: string;
guidAcknowledged: boolean;   // false → old Tome, use legacy flow
```

### 5. Watch loop — `watchApiRecording` (`ApiRecording.ts:138-206`)

- When `guidAcknowledged`: poll `recordingSessionStatus(baseUrl, guid)` instead of the global `/status`. `state === "recording"` → keep waiting; anything else → release the mic lock and hand off to `waitAndLink` exactly as today. `"unknown"` (404) counts toward `MAX_STATUS_FAILURES` like a failure (Tome restarted mid-recording — the session is gone; fall through to `waitAndLink`, whose fallback chain will do its best).
- When not acknowledged: current global-status behavior, unchanged.
- Keep the `startedAt` drift re-anchor; the per-guid response carries `startedAt` too.

### 6. Link tail — `waitAndLink` (`ApiRecording.ts:290-408`)

Replace the completion-poll + file-location logic with a **match ladder**; steps only fall through, never race each other:

1. **Per-guid status poll** (when `guidAcknowledged`): poll `recordingSessionStatus` at the existing `POLL_INTERVAL_MS`/`MAX_POLL_ATTEMPTS` cadence until `state` is `complete` or `failed`.
   - `complete` → resolve the vault file from `transcriptFilename` (the transcript folder is known from `RecordingInfo.transcriptFolderPath`; poll `vault.getAbstractFileByPath` briefly — up to the existing 15×1s window — because Tome writes outside Obsidian and the index may lag). Before linking, read the file's frontmatter and **verify `session_guid` matches**; a mismatch means a stale/foreign file — fall through to step 2.
   - `failed` → surface a `Notice` with the error, clear `CardUiState`, remove the persisted in-flight record, and stop. Do **not** fall through to filename scanning (there is no transcript to find).
   - `unknown` (404 after previously being acknowledged) → fall through to step 2.
2. **Frontmatter guid scan**: scan `transcriptFolderPath` for `.md` files whose cached frontmatter `session_guid` equals the guid (use `metadataCache.getFileCache`; fall back to a raw `vault.adapter.read` of the frontmatter block for files the cache hasn't indexed yet). This catches Tome-crash/orphan-refinalize cases because Tome writes the guid at session *start*.
3. **Legacy heuristics** (unchanged, and the only path when `guidAcknowledged` is false): expected-path poll then `findNewestFile` prefix scan (`ApiRecording.ts:322-344`).

On a successful match (any rung):

- `enrichTranscriptFrontmatter` (`ApiRecording.ts:261-287`): add `session_guid` to the preserved-keys set (alongside `recording`/`voiceprints`) when Tome wrote it; **write it explicitly** when matched via rung 3 with a guid in hand (legacy Tome won't have stamped it — stamping here upgrades the file for future tooling).
- Note frontmatter update (`ApiRecording.ts:381-384`): keep writing `transcript` + `pipeline_state`; `session_guid` is already on the note from start.
- Remove the persisted in-flight record (§7).

Keep the `linkingInProgress` note-path guard as-is.

### 7. Restart recovery — persisted in-flight state

**Storage:** a new top-level field in the plugin settings object persisted by `persistData` (`main.ts:663-664`):

```ts
activeApiRecordings: Array<{
  sessionGuid: string;
  notePath: string;                // best-effort; re-resolve on load, tolerate rename via guid-on-note scan
  transcriptFolderPath: string;
  suggestedFilename: string;
  startedAtIso: string;            // for expiry
  guidAcknowledged: boolean;
}>
```

Write on successful start; remove on successful link or on `failed`. Deliberately **kept** on the not-ready timeout and transcript-not-found paths, even though those show a user-visible warning — a later restart can then still retry the link (implementation improvement over the original "remove on user-visible give-up" rule). This is bookkeeping, not user settings — keep it out of the settings UI.

**Reconciliation on plugin load** (new `reconcileApiRecordings()` called from `onload` after `resetApiRecordingWatchers()`, `main.ts:143`; run non-blocking):

For each persisted entry:

1. Drop entries older than 24h (stamp a `Notice` naming the note so the user knows to use the unlinked flow).
2. Re-resolve the note: `getAbstractFileByPath(notePath)`; if missing, scan the vault for a note whose frontmatter `session_guid` matches (handles rename-while-closed).
3. Query `recordingSessionStatus(baseUrl, guid)`:
   - `recording` → rebuild `RecordingInfo`, `cardUi.setRecording(...)`, restart `watchApiRecording`. (Meeting metadata fields on `RecordingInfo` that only matter for enrichment can be re-derived from the note frontmatter; anything unavailable is acceptable as blank — enrichment already tolerates missing fields.)
   - `transcribing` → spawn `waitAndLink` directly (skip the watch loop).
   - `complete` → run the match ladder immediately and link.
   - `failed` → `Notice` + drop the entry.
   - `unknown` / Tome unreachable → try match-ladder rung 2 (frontmatter guid scan) once; if it hits, link; otherwise leave the entry for the next load (Tome may simply not be running yet) unless expired.

Guard: reconciliation must respect `linkingInProgress` and never double-start a watch for a guid that already has one. (The originally proposed module-level `Set<string>` of reconciled guids was not implemented: reconcile runs exactly once at load, before any user-started session can share a guid, so the set would be dead code. Revisit if reconcile is ever re-run mid-session.)

### 8. Unlinked-recordings flow — `src/services/ApiUnlinkedProvider.ts`

- `findUnlinked` (lines 43-74): no filter change needed (guid presence doesn't imply linked), but surface `session_guid` in the returned candidate metadata.
- **Auto-suggest**: when a candidate transcript's `session_guid` matches a meeting note's `session_guid` (the note was stamped at start but linking failed), pre-select / rank that note first in the pick modal.
- `linkToNote` rename path (lines 76-167, sidecar rename at 225-254): frontmatter travels with the file, so `session_guid` survives the rename automatically. Verify only that the enrichment step preserves the key (same preserved-keys set as §6).

### 9. Voiceprint sidecar — `src/services/VoiceprintEnroller.ts`

`VoiceprintSidecar` interface (lines 56-68) gains optional `sessionGuid?: string`. No validation change (schema stays 1; `model` check unchanged). Not consumed yet — carried for future tooling (e.g. WhisperOrg).

### 10. Out of scope

- `MeetingCard` pill logic, `confirmIfServiceRecording`, MacWhisper flow: untouched.
- Sending `calendar_event_id` through to Tome (the field already exists in `meetingContext` and Tome may start persisting it — that's Tome-side, no WhisperCal change).
- Removing the legacy heuristics — future release, after both apps have soaked.

## Testing checklist (manual — no test suite exists)

Deploy Tome first, then WhisperCal (`main.js`, `manifest.json`, `styles.css` → `~/SDA/.obsidian/plugins/whisper-cal/`).

1. **Happy path:** record from a card → stop → transcript links; note + transcript frontmatter both carry the same `session_guid`; sidecar JSON carries `sessionGuid`.
2. **Back-to-back:** stop meeting A, immediately start meeting B while A is `transcribing`. A's card must link A's transcript and B's card B's — verify each `waitAndLink` polls its own guid and neither cross-adopts.
3. **Same-basename collision:** two notes with identical basenames in different folders, recorded back-to-back — previously ambiguous, must now link correctly.
4. **Restart mid-recording:** start recording, quit Obsidian, relaunch → card shows recording again, stop works, links.
5. **Restart during transcribing:** stop recording, quit Obsidian before link completes, relaunch → transcript links on load.
6. **Old-Tome fallback:** run the pre-change Tome build → `/start` echo missing → whole flow degrades to legacy behavior, still links.
7. **Tome restart mid-session:** kill Tome while recording → watch loop fails over, `waitAndLink` rung 2/3 attempted, user gets a sane notice; unlinked flow still available.
8. **Note rename mid-recording:** rename the meeting note while recording (existing live-`TFile` handling) — link still lands on the renamed note; `session_guid` still matches.
9. `npm run build` and `npm run lint` clean.
